### **2. 狀態轉換流程圖 (State Transition)**

這套邏輯可以拆解為三個主要的狀態流向，這對於撰寫程式碼中的 switch 判斷或 if-else 非常有幫助：

### **路徑一：高風險事件 (強制干預)**

1. 偵測事件 $\rightarrow$ **發送 Push Notification**。
2. 守護者狀態更新為 High Risk。
3. 系統監聽守門人端：直到 onClick(Assisted) 觸發。
4. 狀態回歸 Safe $+$ 寫入 Event Log。

### **路徑二：中風險事件 + 尋求協助**

1. 偵測事件 $\rightarrow$ 守護者點擊「傳送系統通知」。
2. 守護者狀態更新為 Pending。
3. 系統監聽守門人端：直到 onClick(Assisted) 觸發。
4. 狀態回歸 Safe $+$ 寫入 Event Log。

### **路徑三：中風險事件 + 自行處理**

1. 偵測事件 $\rightarrow$ 守護者點擊「撥打 165」。
2. 狀態維持 Safe（不切換風險態）。
3. 動作完成 $\rightarrow$ 立即寫入 Event Log。

### **3. 技術開發建議 (針對 React / React Native)**

在實作這套邏輯時，建議注意以下幾點：

- **狀態同步 (Real-time Sync)**：由於守護者與守門人是不同裝置，建議使用 **Firebase Realtime Database** 或 **Firestore**。當守門人點擊按鈕時，直接修改後端的 status 欄位，守護者端透過監聽 (Listener) 來即時把狀態切回「安全」。
- **欄位設計**：
  - status: 可用 Enum 定義 ['SAFE', 'HIGH_RISK', 'PENDING']。
  - isHandled: 布林值，用來判斷是否要把事件塞進「近期事件」列表中。
- **推播通知 (Expo Notifications)**：高風險事件觸發時，後端或 Cloud Functions 應立即發送 priority: 'high' 的通知給守門人。
